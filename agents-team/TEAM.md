# 共享工作流引擎 (Shared Workflow Engine)

## 1. 核心原则 (Core Principles)

| 原则 | 说明 |
|------|------|
| **单一事实源** | 项目状态以 PM 提交的证据为准，OCA 审计确认后才生效 |
| **证据驱动** | 任何状态流转必须有"书面证据"（报告/日志/确认函），禁止口头变更 |
| **禁止跳跃** | 必须严格按顺序执行阶段，不可跳过门禁 |
| **防幻觉机制** | 未收到明确证据前，默认状态为"未通过" |

---

## 2. 状态机与门禁 (State Machine & Gates)

项目必须严格按以下状态流转，**禁止跳跃**。每个状态转移必须有**书面证据**。

| 当前状态 | 下一状态 | 转移条件 (门禁) | 责任人与证据 |
| :--- | :--- | :--- | :--- |
| **PLANNING** | **DESIGNING** | OCA 批准 PRD 与排期 | OCA 指令 + PM 任务拆解表 |
| **DESIGNING** | **DEVING** | **UX 设计稿定稿** 🔑 | **UI 交付物** (必须包含标注与状态说明) |
| **DEVING** | **INTEGRATING** | FE 与 BE 完成编码 | FE/BE 代码提交记录 |
| **INTEGRATING** | **TESTING** | **联调自测通过** 🔑 | **PM 确认书** (必须包含 FE/BE 双方声明) |
| **TESTING** | **REGRESSION** | 首轮 Bug 修复完成 | QA Bug 列表 (无 P0/P1 遗留) |
| **REGRESSION** | **RELEASE** | **全量回归 100% 通过** 🔑 | **QA 签字报告** (必须包含通过率 100% + 视觉验收) |
| **RELEASE** | **CLOSED** | OCA 批准发布 | OCA 发布指令 + 用户确认 (如需) |

🔑 **关键门禁：** 必须严格检查，未通过不得进入下一阶段

## 3. 关键流程执行规范 (Critical Workflow Standards)

### 3.1 设计冻结门禁 (Design Freeze Gate)

| 项目 | 说明 |
|------|------|
| **触发** | PM 确认 PRD 细节无误 |
| **动作** | UI 输出高保真设计稿，PM 确认符合商业目标 |
| **通过标准** | 设计稿标注清晰，状态完整（正常/异常/交互），PM 签字确认 |
| **禁止** | 若设计稿未定稿，FE **不得** 开始界面编码 |
| **防幻觉** | PM 不得在未收到 UI 交付物时，擅自声称"设计完成" |

### 3.2 联调门禁 (Integration Gate)

| 项目 | 说明 |
|------|------|
| **触发** | FE 与 BE 均声明"编码完成" |
| **动作** | PM 必须询问 FE 与 BE："是否已完成接口联调自测？是否有阻塞性 Bug？" |
| **通过标准** | FE 与 BE 均回复"是" |
| **禁止** | 若任一方未确认，PM **不得** 通知 QA 介入测试 |

### 3.3 回归与视觉门禁 (Regression & Visual Gate)

| 项目 | 说明 |
|------|------|
| **触发** | QA 完成所有 Bug 修复验证 |
| **动作** | QA 必须执行**全量测试用例** + **视觉还原度检查** |
| **通过标准** | 功能通过率 100% **且** 视觉偏差在允许范围内（依据 UI 标准） |
| **禁止** | 若视觉还原度不达标，视为 Bug，不得发布 |

## 4. 沟通闭环机制 (Communication Closed Loop) 🔴 新增

**基于 AI News Pulse 项目复盘（2026-03-11）的核心改进**

### 4.1 重要通知标准流程

```
1. 发送通知（飞书/微信）
   - 标题：【重要】+ 内容摘要
   - 正文：背景 + 行动项 + 截止时间
   - 结尾：**收到请回复，30 分钟内未回复将电话通知**

2. 确认接收
   - 接收方：回复"收到"或提出问题
   - 发送方：记录确认时间，更新追踪表

3. 执行反馈
   - 接收方：完成后回复"已完成"或"遇阻，原因 XXX"
   - 发送方：更新状态看板

4. 超时升级
   - **20 分钟未回复 → 上报 OCA**
   - **30 分钟未回复 → 上报老大（胡宇辰）**
   - 1 小时未完成 → 启动备选方案
```

### 4.2 汇报规范

| 要求 | 说明 |
|------|------|
| **风险前置** | 汇报时把风险项放在最前面，不要藏在末尾 |
| **15 分钟上报** | 发现风险后 15 分钟内上报 OCA，超时自动升级 |
| **书面证据** | 所有状态流转必须有"书面证据"（报告/日志/确认函） |

### 4.3 环境检查

**任务开始前必须确认**:
- 设备情况：手机/电脑/网络
- 可用时间：最晚等待时间窗口
- 备选方案：Plan B（如 Railway 失败→Render）

## 5. 技术前置评审 (Technical Review) 🔴 新增

### 5.1 评审流程

```
1. 需求评审前 24 小时
   - BE 填写《技术可行性评估表》
   - Arch 评审架构方案
   - FE 评估实现难度

2. 评审会议（30 分钟）
   - BE 讲解接口设计
   - FE 确认字段命名规范
   - QA 提出测试关注点
   - 输出《接口定义文档》（Swagger/Apifox）

3. 部署配置验证
   - Arch 编写部署脚本
   - BE/FE 在测试环境验证
   - 输出《部署验证报告》
```

### 5.2 接口定义规范

| 要求 | 说明 |
|------|------|
| **字段命名统一** | 前后端使用相同字段名（如 `url` 而非 `link`） |
| **响应格式统一** | 统一使用 `{success, data, message}` 格式 |
| **文档自动化** | 使用 Swagger/OpenAPI，代码变更自动更新文档 |
| **变更同步** | 接口变更必须在同步群@所有人，并附变更说明 |

### 5.3 技术选型偏好 🔴 新增

**核心原则：优先使用 Java 技术栈**

| 层级 | 首选技术 | 备选技术 | 说明 |
|------|----------|----------|------|
| **后端语言** | 🟢 Java (17+) | Node.js, Python | 优先使用 Spring Boot |
| **前端框架** | 🟢 Vue 3 / React | Angular | 根据团队熟悉度选择 |
| **数据库** | 🟢 MySQL / PostgreSQL | MongoDB | 关系型数据库优先 |
| **缓存** | 🟢 Redis | - | 标准配置 |
| **消息队列** | 🟢 Kafka / RocketMQ | RabbitMQ | Java 生态优先 |
| **部署平台** | 🟢 Docker + K8s | Render, Railway | 容器化优先 |
| **CI/CD** | 🟢 Jenkins / GitLab CI | GitHub Actions | Java 项目经验证 |
| **监控** | 🟢 Prometheus + Grafana | - | 标准配置 |
| **日志** | 🟢 ELK Stack | - | Java 集成成熟 |

**Java 技术栈优势**:
- ✅ 类型安全，减少运行时错误
- ✅ 生态成熟，企业级支持
- ✅ 性能优秀，适合高并发场景
- ✅ 团队熟悉度高，维护成本低
- ✅ 招聘容易，人才储备充足

**例外情况**（可使用非 Java 技术）:
- 快速原型验证（可用 Node.js/Python）
- 前端项目（JavaScript/TypeScript）
- 数据科学/机器学习（Python）
- 特殊场景（如实时性要求极高的 C++）

## 5. 防幻觉检查清单 (Anti-Hallucination Checklist)

在生成任何状态报告前，**必须自问**以下问题：

| # | 问题 | 检查标准 |
|---|------|----------|
| 1 | 我是否有具体的证据（报告/日志/声明）支持当前状态？ | 无证据 = 未通过 |
| 2 | 我是否跳过了某个门禁（如未联调直接测试）？ | 禁止跳跃 |
| 3 | 我的数据（如通过率）是精确的吗？ | 禁止使用"大概"、"可能" |
| 4 | 我是否越级汇报了？ | PM 不得直接联系用户，OCA 不得直接指挥团队 |
| 5 | 设计稿是否已定稿？ | FE 不得无稿开发 |

## 7. 异常处理 (Exception Handling)

| 异常场景 | 处理流程 |
|----------|----------|
| **设计变更** | 若开发中需变更设计，UI 必须评估影响，PM 确认是否延期 |
| **视觉偏差** | 若 FE 实现与设计稿偏差过大，QA 有权驳回并要求修复 |
| **流程违规** | 若发现 FE 无稿开发，PM 必须立即制止 |
| **证据缺失** | OCA 必须将状态回滚至上一阶段，要求 PM 重新收集证据 |
| **风险预警** | PM 必须立即发送 `[风险预警]`，不受状态限制，24 小时内上报 OCA |

# 协作规范 (Collaboration Standards)

## 沟通规范

| 场景 | 规范 |
|------|------|
| **需求变更** | PM 发起 → 全员评审 → Arch 更新文档 → FE/BE 评估工时 |
| **接口变更** | BE 发起 → 更新 Swagger → 通知 FE/QA → 同步修改 |
| **Bug 修复** | QA 提交 → FE/BE 修复 → QA 全量回归 → 关闭 Bug |
| **风险预警** | 发现人 → PM → OCA（重大风险 24 小时内上报） |
| **设计变更** | UI 评估影响 → PM 确认是否延期 → 通知 FE/QA |

## 质量门禁

| 检查点 | 标准 | 负责人 |
|--------|------|--------|
| **设计冻结** | 设计稿标注完整，状态说明清晰 | UI |
| **提测准入** | FE/BE 联调自测报告完整 | QA |
| **发布准入** | 回归测试 100% 通过 + 视觉验收达标 | QA |
| **代码审查** | 核心功能必须 Code Review | Arch |
| **性能指标** | 响应时间、错误率符合 SLA | Arch |
